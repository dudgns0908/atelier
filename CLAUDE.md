# CLAUDE.md — atelier maintainer manual

This file is loaded when working **on atelier itself** — i.e., when a maintainer cds into the plugin directory to develop, fix, or extend atelier.

> **For atelier USERS**: this file does NOT govern your project's behavior. Your project's `CLAUDE.md` is generated by Technical Writer at STEP 5 of `/atelier:init-project` from `docs/templates/project-claude-template.md`. That generated file is the Conductor's runtime manual for your project, not this one.

## What atelier is

`atelier` — universal multi-agent project harness for Claude Code. Run any project, on any stack, through a structured AI team that handles discovery, design, roadmap, PR-reviewed execution, and living documentation.

## Atelier development — universal principles

These apply both to atelier-internal work AND to user projects, so they live in source docs (single source of truth) and are referenced from agent personas:

- **Coding principles** (naming, language, simplicity, critical thinking, interview discipline): `docs/process/coding-principles.md`
- **Living Documentation** (docs are load-bearing, must be updated in same PR): `docs/process/living-documentation.md`
- **Single Source of Truth**: each fact in one canonical doc; references, never duplicates.

When developing atelier, honor these principles for atelier's own artifacts too.

## Hard rules for atelier-itself work

1. **Real job titles only** — for any new agent, LinkedIn test. No invented labels.
2. **No `PM` abbreviation** — `Product Manager` and `Project Manager` always full names.
3. **Hooks remain bypass-resistant** — when modifying `hooks/hooks.json`, ensure no new loophole.
4. **Apache 2.0 + bundled `NOTICE.md` must remain intact** — bundled `skills/skill-creator/` is upstream-licensed.
5. **`bin/atelier-validate` must pass before PR** — structural validation is the minimum bar.
6. **Living Documentation applies to atelier itself** — change a hook? Update `docs/process/coding-principles.md` interview-discipline references? Then update them in the same PR.

## Merge policy (atelier-itself ≠ user-project)

The "Conductor never self-merges" rule from `docs/templates/project-claude-template.md` Hard Rule #1 applies to **user-project Conductors** (the runtime manual generated by Technical Writer at STEP 5). It does **not** govern atelier-itself maintenance.

For atelier-itself contributions, the **3-tier policy** in `CONTRIBUTING.md` § Submitting Changes applies. Every change goes through a feature branch + PR (no direct commits to `main` / `develop`). Review depth varies by tier:

| Tier | Examples | Review depth |
|---|---|---|
| **L1 — Trivial** | typo, logo refresh, single-line doc | Single approve. |
| **L2 — Substantive docs / templates** | process rule, ROADMAP update, agent persona refinement | Solo single-voice 3-lens template (`[Code Quality]` / `[Architecture]` / `[Requirements]`). |
| **L3 — Hooks / agents new / skills new / `bin/*` / `.claude-plugin/` | hook regex change, new skill, atelier-* helper change | Mandatory 3-lens (solo single-voice OR 3 independent when team grows) + regression test if hook touched. |

Pre-push regression checks (`tools/git-hooks/pre-push`) run **locally** — atelier-validate + bash -n + jq -e + shellcheck + sanity checks. **No GitHub Actions CI by default** (aligned with STEP 0.5 default `I.b CI = None`). Set up once after clone:

```bash
git config core.hooksPath tools/git-hooks
```

What "self-merge" means in atelier's design (from `project-claude-template.md`):
- **Forbidden**: Conductor merges *without* the 3-lens gate ever being applied.
- **Allowed**: Conductor (or maintainer) executes the forge merge command **after** the gate is satisfied. The merge itself is part of Conductor's normal job — it's the gate that's mandatory, not the merger's identity.

Direct `git merge feature/*` is hook-blocked (hooks/hooks.json) regardless of policy — always use the forge merge path so `atelier-check-reviews` and `atelier-post-merge` fire.

### When acting as solo maintainer

Two-step pattern:

1. **Review** — post one PR comment with three explicit blocks:
   ```
   APPROVE — solo-maintainer review per CONTRIBUTING.md (single voice, three lenses).

   [Code Quality]
   - <findings>

   [Architecture]
   - <findings>

   [Requirements]
   - <findings>
   ```
2. **Merge** — `gh pr merge <N> --merge --delete-branch` (atelier's git-flow.md default = merge commit; preserves per-commit history).

This is auditable, honest, and matches atelier's spirit (gate must be visibly applied).

## How to develop on atelier

- Read `CONTRIBUTING.md` first.
- Run `bin/atelier-validate` before any PR.
- Use atelier on atelier (dogfood) for substantial changes — see CONTRIBUTING.md "Dogfooding" section.
- Per-version migration notes go in `meta/version-history.md`, not in any CHANGELOG (atelier ships no CHANGELOG.md).
- **No `examples/` directory by design.** atelier-on-atelier (this very repo's `docs/`, `agents/`, `.claude-plugin/`) is the canonical reference example. See README "What atelier looks like in practice" pointer.
